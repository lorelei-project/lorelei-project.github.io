# Thunking System

The Lorelei thunking system consists of guest and host **Thunk Libraries** and a **Runtime**.

## Thunk Libraries

The **Thunk Libraries** serve as the **ABI conversion layer** between guest and host architectures.

Each guest shared library has its corresponding **Guest Thunk Library (GTL)**, **Host Thunk Library (HTL)**, and target host library.

Consider translation from x86_64 to ARM64. For an x86_64 shared library `libX`, its GTL uses the same filename as `libX`, and exports exactly the same functions and variables. Library hijacking is achieved by setting `LD_LIBRARY_PATH` so that the guest dynamic linker (`ld-linux-x86_64.so`) in the emulated environment loads the GTL instead of `libX`. In this way, all calls to `libX` from the guest executable and other unhijacked shared libraries are handled by the GTL as a proxy, while other modules remain unaware.

GTL is compiled with an x86_64 C++ compiler, and HTL is compiled with an ARM64 C++ compiler.

## Thunk Procedure

Each function and function pointer type (i.e., callback) has its own `Guest Thunk Procedure (GTP)` and `Host Thunk Procedure (HTP)`. Each GTP and HTP has three phases: `Entry`, `Caller`, and `Exec`, represented by a `ProcFn` or `ProcCb` record with a static `invoke` method in C++ source code.

For general plain functions, in GTL,
- `Entry` directly calls `Caller`;
- `Caller` packs the arguments and calls `Exec`;
- `Exec` sends the packed arguments to the host environment.

And in HTL,
- `Entry` unpacks the arguments and calls `Caller`;
- `Caller` directly calls `Exec`;
- `Exec` calls the real function in host library.

`Entry` and `Caller` code varies depending on function signatures and behavior, so they are generated by the [`Thunk Library Compiler`](../tools/thunk-library-compiler.md) or implemented manually. `Exec` code is the same for all functions and is implemented in Lorelei headers.

For a function, use the `ProcFn<F, Direction, Phase>` template to define its GTP and HTP. For a callback, use `ProcCb<T, Direction, Phase>`.

```c++
namespace lore::thunk {

    template <auto F, Direction D, Phase P>
    struct ProcFn;

    template <class T, Direction D, Phase P>
    struct ProcCb;
    
}
```

### Example: `memcpy`

Take `memcpy` in `libc.so` as an example. The function signature is:

```c++
void *memcpy(void *dst, const void *src, size_t n);
```


The GTL source code will be:

```c++
namespace lore::thunk {

    template <>
    struct ProcFn<::memcpy, G2H, Caller> {
        static void *invoke(void *dst, const void *src, size_t n) {
            void *ret;
            void *args[] = { &dst, &src, &n }; // pack arguments
            ProcFn<::memcpy, G2H, Exec>::invoke(args, &ret, nullptr);
            return ret;
        }
    };

    template <>
    struct ProcFn<::memcpy, G2H, Entry> {
        static void *invoke(void *dst, const void *src, size_t n) {
            void *ret;
            ret = ProcFn<::memcpy, G2H, Caller>::invoke(dst, src, n);
            return ret;
        }
    };

}
```
- `ProcFn<::memcpy, G2H, Entry>::invoke` will be exported as `memcpy`.
- `ProcFn<::memcpy, G2H, Caller>::invoke` packs the arguments.
- `ProcFn<::memcpy, G2H, Exec>::invoke` sends the packed arguments to the host environment.

And the HTL source code will be:

```c++
namespace lore::thunk {

    template <>
    struct ProcFn<::memcpy, G2H, Caller> {
        static void *invoke(void *dst, const void *src, size_t n) {
            return ProcFn<::memcpy, G2H, Exec>::invoke(dst, src, n);
        }
    };

    template <>
    struct ProcFn<::memcpy, G2H, Entry> {
        static void invoke(void *args[], void *ret, void *metadata) {
            auto &dst = *(void **) args[0];
            auto &src = *(const void **) args[1];
            auto &n = *(size_t *) args[2];
            auto &r = *(void **) ret;
            r = ProcFn<::memcpy, G2H, Caller>::invoke(dst, src, n);
        }
    };

}
```
- `ProcFn<::memcpy, G2H, Entry>::invoke` will unpack the arguments.
- `ProcFn<::memcpy, G2H, Exec>::invoke` will call the real `memcpy` in host library `libc.so`.

### Example: `compare` callback

Take `qsort` in `libc.so` as an example. The function signature is:

```c++
void qsort(void *base, size_t nmemb, size_t size,
           int (*compar)(const void *, const void *));
```

There is a callback `compar` in `qsort`, which makes `qsort` a non-plain function. The thunk library must provide a callback thunk for `compar`.

For the callback `compar`, the GTL source code will be:

```c++
namespace lore::thunk {

    using PFN_compar = int (*)(const void *, const void *);

    template <>
    struct ProcCb<PFN_compar, H2G, Caller> {
        static int invoke(void *callback, const void *a, const void *b) {
            int ret;
            ret = ProcCb<int, G2H, Exec>::invoke(callback, a, b);
            return ret;
        }
    };

    template <>
    struct ProcCb<PFN_compar, H2G, Entry> {
        static int invoke(void *callback, void *args[], void *ret, void *metadata) {
            auto &a = *(const void **) args[0];
            auto &b = *(const void **) args[1];
            auto &r = *(int *) ret;
            r = ProcCb<PFN_compar, H2G, Caller>::invoke(callback, a, b);
        }
    };

}
```

The HTL source code will be:

```c++
namespace lore::thunk {

    template <>
    struct ProcCb<PFN_compar, H2G, Caller> {
        static int invoke(void *callback, const void *a, const void *b) {
            int ret;
            void *args[] = { &a, &b }; // pack arguments
            ProcCb<int, G2H, Exec>::invoke(callback, args, &ret, nullptr);
            return ret;
        }
    };

    template <>
    struct ProcCb<PFN_compar, H2G, Entry> {
        static int invoke(const void *a, const void *b) {
            LORETHUNK_INIT_GCB(callback); // defines `void *callback`
            int ret;
            ret = ProcCb<PFN_compar, H2G, Caller>::invoke(callback, a, b);
            return ret;
        }
    };

}
```

## Callback Trampoline

For `qsort`, we need to handle `compar` specially, because the host library cannot directly call a guest function address.

It is necessary to replace `compar` with the address of `entry` in the following trampoline block:

```asm
saved_callback:
    .qual <compar>

entry:
    ldr x16, [x16, #-8] ; loads saved_callback to x16
    ldr x17, 8          ; loads target to x17
    br x17              ; jumps to target

target:
    .quad <HTP Entry of PFN_compar>
```

The HTP of `qsort` should be:

```c++
namespace lore::thunk {

    // This function is for conceptual demonstration only and is
    // more complex in the actual implementation.
    template <class T>
    T allocCallbackTrampoline(T callback);

    template <>
    struct ProcFn<::qsort, G2H, Entry> {
        static void invoke(void *args[], void *ret, void *metadata) {
            auto &base = *(void **) args[0];
            auto &nmemb = *(size_t *) args[1];
            auto &size = *(size_t *) args[2];
            auto &compar = *(PFN_compar *) args[3];
            auto &r = *(int *) ret;
            r = ProcFn<::qsort, G2H, Caller>::invoke(
                base, nmemb, size, allocCallbackTrampoline(compar)
            );
        }
    };

}
```

The helper `allocCallbackTrampoline` allocates a trampoline block for `compar`, stores `compar` in the `saved_callback` field and `ProcCb<PFN_compar, H2G, Entry>::invoke` in the `target` field, and returns the address of `entry`.

If we apply this workaround, the `LORETHUNK_INIT_GCB` macro in `ProcCb<PFN_compar, H2G, Entry>::invoke` loads `compar` from register `x16` and stores it in `callback`. HTL should be compiled with the `-ffixed-x16` option.

`qsort` is relatively simple, and its HTP code can be generated automatically. However, for many functions, we cannot retrieve all function pointers from argument lists at compile time. In such cases, the host library must be rebuilt with the help of the [`Host Library Rewriter (HLR)`](../tools//host-library-rewriter.md).

## Runtime

The Lorelei runtime is a shared library that consists of **Guest Runtime** and **Host Runtime** components on the guest and host sides. These runtimes maintain Lorelei runtime data, interact with the emulator, and provide runtime APIs to thunk libraries.
